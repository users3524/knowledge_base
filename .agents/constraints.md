# 知识库维护约束

## 内容与结构

- 先搜索现有内容，再决定新建文档还是补充已有文档。
- 每个主题尽量保留一个权威入口，避免复制出多份长期分叉的说明。
- 保持现有一级目录的职责边界；无法明确归类时，先在最接近的现有目录中处理。
- 新增或移动 Markdown 文档后，同步更新受影响的 `README.md` 索引和相对链接。
- 文件名应清晰、稳定；不要仅为统一风格而批量重命名已有文件。

## 编辑原则

- 默认使用 UTF-8 和 Markdown。
- 保留用户未要求修改的内容、格式和工作区变更。
- 修复乱码前先确认源文件编码，不要根据终端中的错误显示直接重写文件。
- 不修改 `.git`、编辑器缓存、构建产物或二进制工程文件，除非任务明确要求。
- 事实不确定时标注假设或来源，不把推测写成确定结论。

## 完成检查

- 检查新增文件是否位于正确主题目录。
- 检查修改过的 Markdown 相对链接是否仍指向存在的文件。
- 检查索引是否能让读者从上一级目录找到新增内容。
- 汇报实际完成的修改和未能执行的验证，不声称未运行的检查已经通过。

## Git 提交日志

- 创建或建议提交信息时，遵循 `01_Frameworks_&_Flows/Git提交日志规范.md`。
- 使用 `<type>(<scope>): <中文摘要>`；`type` 保持 Conventional Commits 英文关键字。
- 范围、摘要和正文使用简体中文，摘要必须说明具体动作，禁止使用“暂存”“补充”等模糊标题。
- 提交前检查暂存区，只提交当前任务相关文件，不夹带用户的其他工作区修改。
