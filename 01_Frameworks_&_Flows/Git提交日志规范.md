# Git 提交日志规范

本仓库采用 Conventional Commits 风格。提交类型使用固定英文关键字，范围、摘要和正文使用
简体中文，以兼顾工具兼容性和中文可读性。

## 基本格式

```text
<类型>[可选范围][可选 !]: <中文摘要>

[可选正文]

[可选脚注]
```

示例：

```text
docs(硬件课程): 新增 Buck 电源环路稳定性笔记

补充环路增益、相位裕度和补偿网络的分析步骤，
并在硬件工程目录中增加入口链接。
```

## 类型

| 类型 | 使用场景 | 示例 |
|---|---|---|
| `feat` | 新增功能、能力或可复用内容 | `feat(ai配置): 安装电路分析 Skill` |
| `fix` | 修复错误、失效链接或错误结论 | `fix(导航): 修复硬件工程目录失效链接` |
| `docs` | 新增或更新知识文档、注释和说明 | `docs(RS-485): 补充终端匹配设计说明` |
| `refactor` | 调整结构但不改变内容含义 | `refactor(知识结构): 合并重复的 CAN 总线说明` |
| `perf` | 提升运行、构建或检索性能 | `perf(搜索): 缩小默认索引扫描范围` |
| `test` | 新增或修改测试、验证用例 | `test(BMS): 补充过压保护测试用例` |
| `build` | 修改构建系统、依赖或工具链 | `build(LaTeX): 更新简历模板编译依赖` |
| `ci` | 修改持续集成配置 | `ci(markdown): 增加失效链接检查` |
| `style` | 只调整格式，不改变语义 | `style(文档): 统一表格对齐格式` |
| `chore` | 维护性杂项，不属于以上类型 | `chore(git): 增加提交信息模板` |
| `revert` | 撤销某个历史提交 | `revert: 撤销“新增自动索引脚本”` |

优先选择最能表达用户价值的类型。知识库新增文章通常用 `docs`；只有新增工具、自动化能力或
可执行工作流时才使用 `feat`。

## 范围

范围用于说明受影响的知识域、模块或工具，建议填写，但不是必填项。

- 使用简短且稳定的中文、英文或常用缩写，例如 `硬件课程`、`BMS`、`导航`、`ai配置`。
- 同一领域保持用词一致，不在 `硬件`、`Hardware`、`HW` 之间随意切换。
- 修改集中于一个明确子模块时，可以使用 `/` 表示层级，例如 `模块整理/数字类低速`。
- 修改跨越多个无明显主次的领域时省略范围，不堆叠范围名称。

## 中文摘要

- 使用“新增、修复、更新、删除、重构、补充、统一”等明确动词开头。
- 说明具体改了什么，避免“暂存”“修改”“补充一下”“更新内容”等模糊描述。
- 控制在一行内，建议不超过 50 个字符，末尾不加句号。
- 不写实现过程、文件数量或无关背景；必要说明放在正文。
- 一次提交只表达一个完整意图，不能用“以及其他修改”打包无关内容。

## 正文与脚注

简单提交可以省略正文。以下情况应填写正文：

- 修改原因仅看摘要无法理解。
- 存在重要设计取舍、兼容性影响或验证边界。
- 同一意图涉及多处关键修改，需要概括说明。

正文使用完整中文句子，重点解释“为什么改”和“改了什么”，不要逐行复述 diff。摘要与正文之间
保留一个空行。

关联问题写在脚注：

```text
Refs: #123
Closes: #456
```

存在破坏性变更时，在类型或范围后增加 `!`，并在脚注中说明迁移方式：

```text
feat(目录)!: 重构知识库一级目录命名

BREAKING CHANGE: 原有目录链接需要更新为新的分类路径。
```

## 提交前检查

1. 使用 `git diff --cached` 确认暂存区只包含本次提交相关内容。
2. 确认类型和范围准确，摘要为具体中文动作描述。
3. 确认没有把临时文件、密钥、构建产物或无关用户修改加入提交。
4. 运行与改动风险相匹配的检查，并在正文中记录重要验证结果或限制。
5. 修改较大且包含多个独立意图时，拆成多个原子提交。

## 推荐与禁止示例

推荐：

```text
docs(电源设计): 补充 Buck 转换器电感选型步骤
fix(目录): 修复嵌入式软件课程大纲链接
refactor(模板): 合并重复的原理图评审检查项
chore(ai配置): 更新知识库维护 Skill 约束
```

禁止：

```text
暂存
补充
update
修改了一些文件
docs: 更新内容以及其他调整
```

## 启用提交模板

仓库根目录提供 `.gitmessage`。需要在当前仓库启用时运行：

```powershell
git config commit.template .gitmessage
```

之后执行 `git commit`，编辑器会自动显示格式和类型提示。以 `#` 开头的提示行不会进入最终提交日志。
