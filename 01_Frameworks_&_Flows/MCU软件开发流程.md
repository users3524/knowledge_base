# MCU软件开发流程与项目白皮书

适用工程：`WRMctrlG47X_bd_HSO`。

当前工程架构：`STM32G473VETx + ST MCSDK 6.4.1 + AD2S1210/RVDT并行总线 + FlashDB/FAL + UART ASPEP/MCP`。

本文的目标不是单纯罗列开发步骤，而是把行业通用的MCU电机控制流程、当前工程真实实现状态、风险缺口和后续重构优先级放在同一份文档中。每个小项都按“理论实现、理论注意、当前实现、方案取舍”分开描述。

## 二、底层架构阶段

### 1. MCSDK移植与工程运行环境

#### 1. 理论是怎么实现的

- 先固定MCU型号、主频、时钟源、PWM频率、ADC时钟、FOC中断周期和中频任务周期，再开始生成或移植MCSDK工程。
- 工程应分层为BSP/HAL或LL驱动层、MCSDK控制内核层、电机参数与标定层、通信对象字典层、应用状态机层。
- 启动文件、中断向量表、链接脚本或Keil Scatter文件需要明确Flash、RAM、参数区、日志区、Bootloader区和升级缓存区。
- 运行环境需要定义裸机调度或RTOS调度边界。FOC高频中断只做强实时控制，通信、参数保存、温度慢采样、日志写入进入低优先级任务。
- 固件应内置固件版本、硬件版本、参数数据库版本、构建时间和Git提交号，方便量产、售后和问题复盘。
- 典型强实时链路应固定为：`TIM1更新/比较事件 -> ADC注入组采样 -> ADC/PWM中断 -> RVDT或角度读取 -> Clarke/Park -> 电流PI -> SVPWM -> TIM1 CCR影子寄存器更新`。
- 若产品需要升级能力，应从一开始规划Bootloader、App、参数区、升级缓存区和故障日志区。实际边界以STM32G473具体Flash容量、擦除页大小和Boot方案为准。
- 固件版本结构建议固定放在只读段，字段至少包含软件版本、硬件PCBA版本、参数数据库版本、构建时间和Git短哈希。

```c
typedef struct {
    uint32_t software_version;
    uint32_t hardware_pcba_version;
    uint32_t parameter_db_version;
    char git_commit_sha[8];
    char build_time[20];
} Firmware_Info_t;
```

#### 2. 理论需要注意哪些

- 时钟树不能只看主频，还要核对TIM1/TIM8计数频率、ADC转换时间、DMA搬运时间和FOC ISR最坏执行时间。
- 链接分区一旦量产后很难调整，Bootloader、参数区和日志区必须预留边界和版本兼容策略。
- 自动生成的MCSDK代码不要随意手改核心模板文件；若必须改，要记录重生成后的合并规则。
- RTOS不是必需品。电机控制里最危险的是高频中断抖动，不是任务数量不够。
- 所有可写参数必须有范围检查、默认值、CRC和参数版本迁移机制。
- 所有新增功能都要标注所属周期和最坏执行时间。FOC中断里每增加一次并行总线读取、滤波或寄存器同步，都要重新评估PWM周期内的算力余量。
- FlashDB/FAL写入和擦除不能进入RUN态强实时路径。内部Flash擦除常以毫秒计，会破坏FOC取指和中断确定性。

### 2. PWM、死区、BKIN与功率驱动

#### 1. 理论是怎么实现的

- 三相逆变器通常使用TIM1/TIM8高级定时器产生中心对齐互补PWM。
- 上下桥臂必须配置死区时间，死区值由驱动芯片传播延迟、MOS/IGBT关断时间、栅极电阻和温度漂移共同决定。
- PWM比较值应通过影子寄存器同步更新，避免一个PWM周期内三相占空比更新不一致。
- BKIN/BKIN2应接外部硬件过流、驱动故障、去饱和或比较器输出，实现不依赖软件的快速封锁。
- 复位、上电、故障和调试暂停时，六路PWM默认态必须落到硬件安全状态。
- 死区时间应由器件关断/开通延迟、上升/下降沿时间和温漂裕量共同决定。工程上可按下面的保守约束估算后，再用示波器验证上下桥互补波形：

$$T_{dead} > (t_{d(off)} - t_{d(on)}) + (t_f - t_r) + T_{margin}$$

#### 2. 理论需要注意哪些

- BKIN是功率安全链路，不应因为调试方便长期关闭。
- 死区过小会直通，死区过大会带来电压畸变、低速转矩脉动和电流过零误差。
- 单纯软件关PWM响应时间取决于中断延迟和代码路径，不能替代硬件过流封锁。
- 高速永磁电机深弱磁故障时，直接高阻关PWM可能导致反电动势经体二极管回灌母线，需要ASC策略。
- BKIN极性、滤波、锁存方式和恢复方式必须与外围驱动电路一致，不能只按默认配置。

#### 3. 当前工程实际是怎么实现的

- TIM1三相互补PWM、死区、六路PWM引脚和低边控制已配置。
- `PE15`有`TIM1_BKIN`配置痕迹，也看到`LL_TIM_SetBreakInputSourcePolarity`等配置。
- 当前关键缺口是`TIM_BDTRInitStruct.BreakState = LL_TIM_BREAK_DISABLE`，说明硬件Break实际没有启用。
- 故障策略主要仍是MCSDK软件路径封锁PWM，未看到高速弱磁场景下的主动短路ASC。

#### 4. 哪种更好，为什么用现在这种

- 从安全角度，更好的方案是启用BKIN硬件刹车，并把外部过流、驱动故障映射到TIM1硬件封锁。
- 当前关闭BKIN可能是为了早期调试避免误触发、极性未确认或硬件故障脚尚未稳定，但这只能作为实验阶段状态。
- 进入功率台架和量产前，应该优先恢复BKIN，并用低压限流台架验证触发极性、封锁时序和故障恢复流程。
- 建议的落地路径是先确认Gate Driver Fault输出极性，再把Break打开，并禁止故障后自动恢复PWM：

```c
TIM_BDTRInitStruct.BreakState = LL_TIM_BREAK_ENABLE;
TIM_BDTRInitStruct.BreakPolarity = LL_TIM_BREAK_POLARITY_HIGH; /* 按硬件实际极性调整 */
TIM_BDTRInitStruct.AutomaticOutput = LL_TIM_AUTOMATICOUTPUT_DISABLE;
```

### 3. ADC采样、ICS电流反馈、母线电压与温度测量

#### 1. 理论是怎么实现的

- 相电流采样应由高级定时器触发ADC注入组，使采样点落在PWM噪声最小的窗口。
- 双电阻、三电阻、单电阻和ICS/隔离电流传感器的采样窗口不同，算法需要按硬件拓扑选择对应PWM Current Feedback实现。
- 母线电压、功率器件温度、电机绕组温度等慢变量可走ADC规则组和DMA，再由中频或低频任务滤波处理。
- 所有模拟量都应换算为物理单位，并带量程、滤波、限幅、开路短路诊断和失效降级策略。
- ADC触发点、采样时间、转换时间、DMA搬运和FOC读取必须做时序闭合验证。
- 中心对齐PWM下，ADC注入触发点通常选在计数器峰值、谷值或避开开关沿的比较事件，目标是让电流采样落在电流纹波和开关噪声最小的窗口。
- FOC周期预算应满足：

$$T_{PWM} > T_{sample} + T_{convert} + T_{read} + T_{angle} + T_{FOC} + T_{CCR\_update} + T_{margin}$$

#### 2. 理论需要注意哪些

- ADC采样点应避开死区、开关沿、反向恢复噪声、母线纹波和运放恢复时间。
- 极端占空比下采样窗口可能消失，低侧采样尤其明显，需要PWM移相、特殊采样窗口或采样重构策略。
- 温度采样不能只做过温阈值，还要识别NTC开路、短路、跳变和温升斜率异常。
- 母线电压滤波不能太慢，否则过压/欠压保护滞后；也不能太快，否则开关纹波导致误保护。
- 电流采样饱和、ADC满量程、偏置漂移和相间增益误差都会直接进入FOC，造成扭矩误差甚至失控。

#### 3. 当前工程实际是怎么实现的

- 当前工程使用`PWMC_ICS_Handle_t PWM_Handle_M1`，属于ICS双电流采样链路。
- ADC1/ADC2注入采样由TIM1触发，用于电流环；规则通道用于母线电压、温度等慢变量。
- MCSDK的`PWMC_CurrentReadingCalibr`已接入偏置校准，`regular_conversion_manager.c`参与慢变量转换管理。
- 母线电压、温度采样和过压/欠压/过温保护已接入MCSDK安全任务。
- 未看到温度线性降额、温升斜率保护、三相电流残差诊断和相间增益标定。
- 附件中建议相电流采样时间至少按ICS运放建立时间保守设置，这一点应结合当前ADC配置、运放带宽和示波器实测噪声再定，不能只追求最短ADC采样周期。

#### 4. 哪种更好，为什么用现在这种

- 既然当前硬件与工程已经走ICS双电流采样，继续使用MCSDK的ICS路径是更好的选择，采样窗口约束比低侧电阻方案小，移植风险也低。
- 理论上三相独立电流采样加完整残差诊断更强，但硬件不一定具备第三相独立通道，软件也需要更复杂的校准流程。
- 当前方案适合先完成闭环控制和参数辨识，但必须补相间增益标定、采样失效诊断和温度降额，才能从“能跑”进入“可量产”。

### 4. 电流零点偏置、增益校准与残差诊断

#### 1. 理论是怎么实现的

- 零点偏置校准应在PWM封锁、功率桥未使能、确认无相电流时执行。
- 连续采样256或512个ADC周期，求平均值，得到运放和ADC链路的静态偏置。
- 运行期每次读取相电流后先减偏置，再进入Clarke/Park变换。
- 增益校准需要已知电流源、工装或低压注入，在不同相位上比较采样通道比例，得到每相Gain Factor。
- 运行期应监测`Ia + Ib + Ic`残差、ADC饱和、采样跳变和通道失联。
- 静态偏置可按下面的均值公式计算：

$$V_{offset} = \frac{1}{N}\sum_{n=1}^{N}ADC\_Raw(n)$$

- 若确实需要运行期慢速偏置跟踪，只能在明确零电流窗口内以极小系数低通更新：

$$V_{offset}(k) = (1-\alpha)V_{offset}(k-1) + \alpha ADC\_Raw(k), \quad \alpha \le 0.001$$

#### 2. 理论需要注意哪些

- 空采样只能校准偏置，不能校准相间增益。
- 若运行中做偏置慢跟踪，必须保证确实处于零电流工况，否则会把真实负载电流当成漂移扣掉。
- 增益校准需要保存到NVM，并带CRC、版本、上下限和恢复默认机制。
- 残差诊断阈值要考虑电流重构误差、滤波延迟和PWM工况，不应设置得过窄。
- 校准流程必须与状态机互锁，防止用户在带载运行时误触发。
- 相间增益标定最好在出厂工装中完成。仅靠上电空采样无法识别采样电阻、运放反馈电阻和ADC通道增益差。
- 运行期残差诊断可用`abs(Ia + Ib + Ic)`作为主指标，并叠加ADC接近满量程、接近零码、突跳和长时间不变等失效特征。

#### 3. 当前工程实际是怎么实现的

- `[已实现]` 电流零点偏置校准已通过MCSDK状态机接入，在`OFFSET_CALIB`状态执行。
- `[未发现实现]` 未看到三相电流增益一致性校准、采样电阻/运放增益标定。
- `[未发现实现]` 未看到运行期`Ia + Ib + Ic`残差诊断和ADC饱和边界诊断的专用实现。

#### 4. 哪种更好，为什么用现在这种

- 当前只做偏置校准是MCSDK默认且最稳妥的基本方案，适合早期闭环运行。
- 更好的量产方案是“每次上电偏置校准 + 出厂增益标定 + 运行残差诊断”三件套。
- 现在先用偏置校准，是因为它不需要工装、风险低、可自动执行；增益标定需要测试设备和产线流程，应作为下一阶段补强。

### 5. RVDT/AD2S1210位置反馈与相序零位对齐

#### 1. 理论是怎么实现的

- 旋变/RVDT系统由激励信号、正余弦反馈、解码芯片或软件解算、角度滤波、速度估算和故障诊断组成。
- AD2S1210这类专用解码芯片可通过并行总线或SPI读取绝对位置、速度和故障位。
- 解算后的机械角度需要按极对数转换为电角度，并统一到MCSDK使用的S16角度格式。
- 相序和极性对齐通常通过锁轴电流或固定电压矢量完成，读取此时传感器角度作为电角度零位偏置。
- 高速下需要考虑采样、解码和总线读取延迟，必要时做速度超前角补偿。
- AD2S1210并行读取需要满足`SAMPLE`、`CS`、`RD`和数据总线建立/保持时间。若读取在FOC中断内完成，应把GPIO翻转、总线读数和角度换算的耗时纳入WCET。
- 高速角度补偿可按固定延迟做一阶超前：

$$\theta_{out} = \theta_{raw} + \omega_e \cdot \Delta t$$

#### 2. 理论需要注意哪些

- RVDT角度方向、三相UVW相序、电角度正方向必须一致，否则电机会抖动、反转或过流。
- AD2S1210的DOS/LOT等故障位必须进入主故障状态机，不能只存在本地变量里。
- 并行总线读取如果放在最高频FOC中断里，要评估读取耗时和中断抖动。
- 零位对齐需要低电压、限电流、限时间，并确保机械负载允许锁轴。
- 对齐结果必须持久化，并带手动重新标定和恢复默认机制。

#### 3. 当前工程实际是怎么实现的

- AD2S1210/RVDT驱动已实现，`main()`中调用`AD2S1210_Init(&RTFPare, 4, 92)`。
- 当前工程重写/覆盖了`ENC_CalcAngle()`和`ENC_CalcAvrgMecSpeedUnit()`，从AD2S1210读取电角度、机械角度和速度，并通过`ENCODER_M1`抽象提供给FOC。
- RVDT相关参数和角度可通过MCP寄存器读写。
- DOS/LOT故障位已读到`RTFPare.Fault_DOSLOT`并影响本地`Move_State`，但未看到上报MCSDK故障状态机的闭环。
- `RVDT_Motor_Alignment()`存在，但未看到在状态机中稳定调用以完成产线式自动零位对齐。
- 更完整的闭环应在中频安全任务中周期读取AD2S1210状态寄存器，并把DOS/LOT映射为位置反馈故障或Break类严重故障，而不是只改变本地变量。

#### 4. 哪种更好，为什么用现在这种

- 当前用AD2S1210硬件解码RVDT作为主位置反馈，对低速、零速、大动态和启动可靠性都优于纯无感方案。
- 比起自己用ADC采Sin/Cos再软件解算，AD2S1210方案更适合当前项目，因为它降低了FOC实时负担并提供硬件诊断位。
- 当前方案需要补齐的是故障闭环和自动零位标定，而不是改成无感主控。
- 推荐优先补的不是更换反馈方案，而是把`Fault_DOSLOT`接入MCSDK故障状态机，并把RVDT零位偏置保存进FlashDB参数表。

## 三、电机特征提取与参数管理阶段

### 1. Rs/Ld/Lq/Ke基础参数辨识（SCC）

#### 1. 理论是怎么实现的

- 定子电阻Rs可通过低速或静止注入已知电压/电流，根据稳态电压电流关系估算。
- d/q轴电感可通过高频注入、阶跃电压响应或小信号扫频，分析电流斜率、幅值和相位。
- 反电动势常数Ke可通过开环拖动、V/F加速到测量点或外部台架反拖测得。
- 辨识任务通常拆成中频状态机和高频采样输出两部分。中频负责阶段切换，高频负责电压注入、采样、滤波和峰值检测。
- 辨识结果应进入电机参数结构，用于电流环PI、弱磁、前馈、MTPA、速度环整定和保护限幅。
- 推荐的闭环链路是：`SCC辨识完成 -> 合理性检查 -> CurrPI_Tune更新电流环PI -> OTT整定速度环 -> 人工确认或台架门禁 -> FlashDB事务保存`。

#### 2. 理论需要注意哪些

- 辨识时电机可能发热、抖动或旋转，必须有限流、限压、限时和急停条件。
- Rs对温度敏感，冷态辨识和热态运行会存在偏差。
- Ld/Lq会随电流饱和变化，单点辨识只能代表某个工作点。
- Ke辨识需要确认速度、极对数、角度方向和电压缩放，否则误差会传导到弱磁和前馈。
- 辨识流程不能和普通FOC闭环同时抢占PWM输出。

#### 3. 当前工程实际是怎么实现的

- 当前由`Src_sensorless\libmp_replace\mp_self_com_ctrl.c`中的SCC框架替代旧`motor_id_fsm`实现。
- SCC覆盖Rs、Ls/Ld/Lq、BEMF/Ke子状态机。
- `mc_tasks.c`中在SCC激活时会走`SCC_HF()`和`SCC_SetPhaseVoltage()`，抢占普通FOC路径。
- `mp_self_com_ctrl.c`中频任务处理Rs、Ls和BEMF阶段切换。
- 旧`motor_id_hftask.c`和`bemf_vf_ctrl.c`存在但不应作为当前主实现判断依据。

#### 4. 哪种更好，为什么用现在这种

- 当前SCC替代库比保留多套旧辨识路径更好，因为它已经接入当前高频任务和MCP寄存器链路。
- 理论上台架离线辨识精度更高，但在线/半在线SCC更适合调试和快速适配多台电机。
- 当前方案应继续保留，但需要增加辨识结果合理性检查、温度记录、失败回退和参数保存事务。
- 附件中把`CurrPI_Tune()`说成未接入，这一点以当前代码复核结果为准：函数已经在SCC完成后和OTT启动前调用；真正缺口是整定结果的确认、回退和完整持久化。

### 2. 电流环PI自整定与速度环OTT自整定

#### 1. 理论是怎么实现的

- 电流环PI通常基于电机Rs、Ld、Lq和目标带宽计算，可采用零极点对消或带宽法。
- 常见公式为`Kp_d = BW * Ld`、`Kp_q = BW * Lq`、`Ki = BW * Rs`，再按MCSDK定点格式量化。
- 速度环自整定可通过阶跃响应估算惯量J、摩擦F，再计算速度环Kp/Ki。
- 自整定后需要写入PID控制器，并可选择保存为参数默认值或当前会话值。
- 速度环参数可按速度区间分段，低速重刚性，高速重稳定和抗共振。
- 电流环目标带宽应显式参数化，并在量化后进行上下限保护：

$$K_{p,d} = \omega_{cc}L_d,\quad K_{p,q} = \omega_{cc}L_q,\quad K_i = \omega_{cc}R_s$$

#### 2. 理论需要注意哪些

- 电流环带宽不能过高，一般要低于FOC采样频率的若干分之一，并留出相位裕量。
- 参数量化后可能出现Kp/Ki过小、过大或饱和，需要上下限和最小有效值。
- 速度环阶跃测试必须限制Iq、速度、时间和机械安全范围。
- 自整定结果不能无条件覆盖人工标定值，应该支持确认、回退和版本记录。
- 多段PID切换要避免参数突变造成速度环输出跳变。

#### 3. 当前工程实际是怎么实现的

- `CurrPI_Tune()`已编译并调用，函数基于SCC结果计算Id/Iq电流环PI，并写入`PIDIdHandle_M1`、`PIDIqHandle_M1`。
- `CurrPI_Tune()`在SCC辨识完成后调用，OTT启动前也会先调用。
- OTT由`mp_one_touch_tuning.c`实现，采用阶跃响应法估算J、F并计算速度环PI。
- OTT可通过MCP寄存器启动/停止，并读取状态、原始Kp/Ki、J、F、带宽等信息。
- 多段速度PID已通过`MOTOR_MULTIPID_ENABLE 1`启用，并在运行中按速度段切换。

#### 4. 哪种更好，为什么用现在这种

- 当前“辨识参数 -> 电流环PI -> OTT速度环 -> 多段PID”的链路是比较合理的，比手动盲调更可重复。
- 理论上最终仍要台架人工确认，因为自动整定不能覆盖机械共振、负载弹性、温升和工艺冲击。
- 现在使用自动整定是好的，因为它能把调试起点拉到可运行范围；量产前应增加参数冻结、回退和测试报告门禁。

### 3. 饱和电感、MTPA与MTPV

#### 1. 理论是怎么实现的

- IPMSM在大电流下Ld/Lq会随Id/Iq变化，形成非线性电感Map。
- 饱和电感可通过大电流扫点、阶跃响应或台架测试得到`Ld(Id,Iq)`、`Lq(Id,Iq)`查找表。
- MTPA根据磁链、电感差和目标转矩，寻找同等转矩下电流最小的Id/Iq组合。
- MTPV用于高速电压受限区，在电压极限下寻找最大可输出转矩轨迹。
- 饱和Map、MTPA表和MTPV表应与电机型号和温度条件绑定保存。

#### 2. 理论需要注意哪些

- MTPA/MTPV对电机类型强相关，SPMSM、IPMSM、同步磁阻和感应电机算法不同。
- 大电流扫点有退磁、过热和机械冲击风险。
- 错误的MTPA表可能比`Id=0`更差，甚至引起电流振荡。
- 弱磁、MTPA和电流/电压圆限幅之间需要统一仲裁，不能各算各的。
- 查找表插值必须处理边界、异常值和参数版本。

#### 3. 当前工程实际是怎么实现的

- 当前未看到MTPA/MTPV实际控制实现，仅见配置标志或注释。
- 当前已具备Rs、Ld/Lq、Ke等基础辨识能力，但没有形成饱和电感二维Map。
- 当前主控制策略更接近标准PMSM FOC、速度/转矩闭环和弱磁控制。

#### 4. 哪种更好，为什么用现在这种

- 如果当前电机更接近表贴式PMSM或效率优化不是第一目标，先用`Id=0 + 弱磁`比贸然上MTPA更稳。
- 对IPMSM或高效率产品，更好的最终方案是补饱和电感Map和MTPA/MTPV。
- 现在不用MTPA/MTPV是合理的阶段性选择，因为当前更紧急的是稳定闭环、保护、参数持久化和弱磁边界验证。

### 4. 死区补偿与逆变器非线性标定

#### 1. 理论是怎么实现的

- 死区补偿根据相电流方向判断电压误差方向，在PWM占空比中补回死区造成的等效电压损失。
- MOS/IGBT导通压降、开关延迟、驱动传播延迟和母线电压都会造成实际输出电压偏离理想SVPWM。
- 可通过空载或低载下已知电压矢量测试，建立“指令电压 -> 实际电流响应”的补偿曲线。
- 低速高精度控制中，逆变器非线性补偿会显著改善电流正弦度和转矩脉动。
- 死区等效电压误差可用相电流方向决定符号，工程近似可写为：

$$V_{phase\_comp} = sign(I_{phase}) \cdot \frac{T_{dead}+T_{on}-T_{off}}{T_{pwm}} \cdot V_{bus}$$

#### 2. 理论需要注意哪些

- 死区补偿依赖电流方向，电流过零处最容易抖动，需要滞环或滤波。
- 补偿过量会引入新的畸变，甚至导致低速振荡。
- 不同母线电压、温度和器件批次下补偿值可能变化。
- 标定曲线应有边界和插值策略，不应在高频中断里做复杂浮点查表。

#### 3. 当前工程实际是怎么实现的

- 当前已配置硬件死区。
- 未看到专门的软件死区补偿、导通压降补偿、开关延迟补偿或逆变器非线性标定曲线。

#### 4. 哪种更好，为什么用现在这种

- 当前先依赖硬件死区和MCSDK基础SVPWM是合理的，能降低早期调试变量。
- 更好的高性能方案是补死区补偿和逆变器非线性标定，尤其用于低速大扭矩、低噪声和高效率场景。
- 建议在电流采样、RVDT、PI和弱磁稳定后再加补偿，否则多个因素叠加会让调试难以归因。

### 5. FlashDB/FAL、EEPROM与参数持久化

#### 1. 理论是怎么实现的

- 电机参数、控制器参数、标定值、PID、弱磁、前馈、保护阈值和故障日志应进入统一参数数据库。
- 参数应包含版本号、长度、CRC、默认值、最小/最大值、单位和权限。
- Flash写入应采用事务机制或双备份页，避免掉电写坏后系统无法启动。
- 高频控制中断不得执行Flash擦写，参数保存应放到低优先级任务。
- 关键参数修改应记录来源、时间戳和写入结果。

#### 2. 理论需要注意哪些

- 内部Flash擦写会阻塞或影响取指，必须避开强实时控制路径。
- 参数结构体升级时要做版本迁移，否则旧参数会被新固件误读。
- 频繁保存会损耗Flash寿命，需做节流和脏标记。
- 故障日志和参数保存最好分区，避免日志刷写破坏控制参数。
- 外部EEPROM寿命更高但速度慢、容量小，适合低频参数，不适合高速黑匣子。

#### 3. 当前工程实际是怎么实现的

- `main.c`中已调用`fal_init()`和`fdb_kvdb_init(&kvdb, "param", "kvdb", NULL, NULL)`。
- `ParamStore_LoadAll()`在启动时加载参数，寄存器命令可触发保存。
- `MotorParam`包含极对数、额定转速、额定电流、Rs、Ld、Lq、Ke、母线阈值、最大电流和多段速度PID。
- 当前参数库实际走内部FlashDB/FAL。
- 24C64 EEPROM/I2C驱动已编译和初始化，但更像备用或遗留路径。
- 弱磁/前馈参数当前保存为0且恢复处标注“restore when implemented”，故障日志/TSDB未接入。

#### 4. 哪种更好，为什么用现在这种

- 当前使用内部FlashDB/FAL比同时维护EEPROM和内部Flash两套参数路径更好，接口统一、调试方便。
- EEPROM可作为后续冗余或特定高频写入参数的备选，但当前不应分散参数来源。
- 更好的下一步是完善FlashDB参数表，把弱磁、前馈、ASC阈值、温度降额、RVDT零位和故障日志纳入统一版本化管理。

## 四、核心闭环控制阶段

### 1. PMSM FOC电流环、速度环与SVPWM

#### 1. 理论是怎么实现的

- FOC核心链路为相电流采样、Clarke变换、Park变换、Id/Iq PI控制、反Park、SVPWM、电压限幅和PWM更新。
- 速度模式下，速度环输出Iq参考；转矩模式下，上层直接给Iq或转矩参考。
- Id通常用于磁链控制，SPMSM常用`Id=0`，弱磁区注入负Id。
- SVPWM要结合母线电压动态限制电压矢量，防止调制过饱和。
- 电流环、速度环、限幅、斜坡和状态机必须在固定周期执行。

#### 2. 理论需要注意哪些

- 电流环是最内环，采样延迟、角度误差和PWM延迟都会降低稳定裕量。
- 速度环输出必须有限幅和斜率限制，否则会把电流环长期推入饱和。
- PI抗积分饱和必须与电压圆、电流圆和弱磁共同设计。
- 编码器/RVDT角度跳变会直接造成dq轴解耦错误，故障应快速闭环。
- 调试时应先跑电流环，再跑速度环，最后做弱磁和动态扰动。

#### 3. 当前工程实际是怎么实现的

- 当前PMSM FOC电流闭环、速度闭环、转矩模式/速度模式、SVPWM/圆限幅等MCSDK基础链路已接入。
- 默认控制模式为速度模式，默认目标转速1000rpm。
- 控制主链路在`mc_tasks_foc.c`、`speed_torq_ctrl.c`、`mc_interface.c`等文件中实现。
- 多段速度PID已启用，运行中按速度段切换。

#### 4. 哪种更好，为什么用现在这种

- 当前使用MCSDK成熟FOC内核是更好的选择，没必要重写Clarke/Park/SVPWM/PID基础算法。
- 自研FOC只有在特殊硬件或特殊控制目标下才值得做，否则验证成本很高。
- 当前应把精力放在参数、保护、反馈、状态机和高级功能补齐，而不是替换基础FOC内核。

### 2. 抗积分饱和、动态限幅与斜坡管理

#### 1. 理论是怎么实现的

- 电流环PI和速度环PI都需要抗积分饱和，常见方式包括积分冻结、条件积分和回算抗饱和。
- 电压矢量应满足`Ud^2 + Uq^2 <= Umax^2`，其中`Umax`由实时母线电压和调制方式决定。
- 电流矢量应满足`Id^2 + Iq^2 <= Imax^2`，弱磁和转矩电流需要统一分配电流圆。
- 速度、转矩、位置目标应经过Ramp或S曲线，避免阶跃命令直接冲击内环。

#### 2. 理论需要注意哪些

- 限幅不是简单截断，限幅后PI积分器若继续累加，会在解除限幅时产生大过冲。
- 弱磁区应优先保证负Id或电压裕量，否则容易失去高速稳定性。
- 速度环斜坡时间过短会表现为电流环饱和，而不一定是速度环参数错误。
- 动态限幅需要考虑温度、母线电压、功率器件能力和电机退磁边界。

#### 3. 当前工程实际是怎么实现的

- MCSDK内已有PID、速度/转矩Ramp、SVPWM和基础电压圆限幅机制。
- OTT中有阶跃与加速阶段管理，多段速度PID也参与不同速度区间控制。
- 未看到完整温度降额、电流残差诊断触发后的动态限幅和ASC联动限幅。

#### 4. 哪种更好，为什么用现在这种

- 当前使用MCSDK基础抗饱和和Ramp机制是正确起点，能快速形成稳定闭环。
- 更好的工程方案是把母线、温度、弱磁、电流圆和故障状态统一纳入限幅仲裁。
- 现在先用MCSDK机制，是因为它和已有状态机、PID结构兼容；后续补仲裁层比改内核更稳。

### 3. 高速弱磁控制

#### 1. 理论是怎么实现的

- 当转速达到基速后，反电动势接近母线电压极限，控制器需要注入负Id削弱等效磁链。
- 弱磁控制通常监测电压利用率或电压矢量幅值，通过弱磁PI输出负Id参考。
- 弱磁必须同时受电流圆、退磁边界、母线电压、速度上限和温度约束。
- 弱磁退出时负Id需要平滑回零，避免母线泵升和转矩突变。
- 弱磁区故障策略应和ASC联动：低速故障可高阻关PWM，高速深弱磁故障需要先评估反电动势和母线吸收能力，必要时进入限时主动短路。

#### 2. 理论需要注意哪些

- 负Id过大可能导致永磁体退磁。
- 弱磁PI过激会导致Id/Iq振荡、电压利用率打摆和高速啸叫。
- 高速减速会产生回馈能量，若母线吸收能力不足，会触发过压。
- 弱磁参数必须通过稳态边界和动态跨基速测试确认。
- 弱磁区急停不能只看当前故障码，还要同时读取机械速度、电压利用率、母线电压、Id/Iq限幅状态和功率级SOA。

#### 3. 当前工程实际是怎么实现的

- MCSDK弱磁控制已接入，包含`FW_M1`、`PIDFluxWeakeningHandle_M1`、`FW_Init()`、`FW_CalcCurrRef()`、`FW_DataProcess()`。
- MCP寄存器可读写`FLUXWK_KP/KI/BUS/BUS_MEAS`等弱磁相关参数。
- 当前FlashDB参数保存中弱磁参数仍不完整，保存/恢复存在留白。
- 未看到弱磁区ASC、温度降额和母线能量处理的联动。

#### 4. 哪种更好，为什么用现在这种

- 当前使用MCSDK弱磁组件比自研弱磁环更好，因为它已经和FOC电流参考、限幅和寄存器接口相连。
- 现在的问题不是弱磁算法不存在，而是弱磁参数持久化、安全边界和高速故障策略不完整。
- 更好的方向是保留MCSDK弱磁，在外侧补退磁限幅、ASC触发条件、母线过压处理和系统测试门禁。
- 推荐把`BaseSpeed`、最大允许负Id、最大ASC电流、ASC持续时间和过压阈值统一放进参数表，避免弱磁、ASC和过压保护各自使用不同边界。

### 4. 前馈解耦

#### 1. 理论是怎么实现的

- PMSM dq轴模型中，d轴和q轴存在交叉耦合项。
- 前馈解耦通过电角速度、Ld、Lq、磁链和电流反馈计算补偿电压。
- 常见形式为补偿`-omega_e * Lq * Iq`和`omega_e * (Ld * Id + psi_m)`等项。
- 前馈可减轻PI负担，提高高速和大动态工况下的电流跟随能力。

#### 2. 理论需要注意哪些

- 前馈强依赖Rs、Ld、Lq、Ke、速度和电流缩放，参数错了会反向补偿。
- 前馈应与电压限幅和弱磁统一处理，不能突破电压圆。
- 低速时前馈收益有限，噪声和量化误差可能更明显。
- 启用前要先确保电机参数辨识和单位换算可靠。

#### 3. 当前工程实际是怎么实现的

- 前馈解耦组件已初始化，并有寄存器读取痕迹。
- `mc_config.h`中`USE_FEEDFORWARD_DECOUPLING`当前为0，FOC计算中相关逻辑被编译开关关闭。
- SCC辨识完成时有计算前馈常数的代码痕迹，但主控制路径未真正启用。

#### 4. 哪种更好，为什么用现在这种

- 当前关闭前馈是保守且合理的，因为电机参数、弱磁参数持久化和高速验证还未完全闭环。
- 更好的性能方案是在SCC参数可信后启用前馈，特别是高速弱磁和突加负载场景。
- 推荐先做参数单位校核和台架对比，再分阶段打开前馈，而不是直接把宏改为1进入量产。

### 5. 无感观测器、RVDT冗余与飞车顺风启动

#### 1. 理论是怎么实现的

- 无速度传感器控制可按速度段选择HFI、SMO、EKF、STO-PLL或反电动势过零检测。
- 零速/极低速PMSM若无位置传感器，常用高频注入HFI；中高速可用SMO、EKF或STO-PLL。
- 有位置传感器系统也可以并行运行无感观测器，用作角度合理性校验和故障降级。
- 飞车启动应先静默观测当前转速和角度，禁止直接锁轴，再无冲击并入FOC。

#### 2. 理论需要注意哪些

- 无感观测器在零速和低速可观测性差，不能替代RVDT的零速绝对角度能力。
- 飞车启动最怕在电机已转时执行固定角度对齐，会产生反向电流冲击。
- 观测器角度和RVDT角度比较要考虑滤波延迟、极对数、方向和电角度周期跳变。
- 无感降级必须定义速度范围和扭矩限制，不能假设所有工况都可无感运行。

#### 3. 当前工程实际是怎么实现的

- STO-PLL无感观测器已编译、初始化，并有Sensorless中频/高频路径。
- 当前默认驱动模式为`MCI_DRIVE_SENSORED`，主线走RVDT/Encoder抽象。
- 飞车/顺风启动只看到Sensorless路径中的BEMF停止检测、RevUp和SwitchOver逻辑。
- 未看到完整“PWM关闭静默观测当前速度角度并无缝并入FOC”的成品级飞车启动流程。
- 未看到HFI、SMO、EKF接入。

#### 4. 哪种更好，为什么用现在这种

- 当前以RVDT为主、STO-PLL为辅是更适合该项目的方案，因为RVDT能覆盖零速、低速和启动角度。
- 纯无感方案对成本更友好，但会牺牲低速和启动可靠性，不适合已经配置RVDT的高可靠驱动器优先级。
- 后续更好的做法是让STO-PLL成为冗余诊断和飞车辅助，而不是替代RVDT主链路。

### 6. 多电机控制内核切换

#### 1. 理论是怎么实现的

- 通用驱动平台可抽象电机控制接口，按电机类型挂载PMSM FOC、BLDC六步、ACIM FOC/DTC等控制内核。
- 每种电机需要独立参数模型、状态机、观测器、保护策略和调试流程。
- 运行时切换电机类型通常不推荐，更多是在编译期或参数配置期选择。

#### 2. 理论需要注意哪些

- 多内核会显著增加测试矩阵和安全风险。
- BLDC六步、PMSM FOC、ACIM矢量控制对反馈、PWM和参数的要求不同。
- 共享寄存器和状态机时，要避免某个电机类型写入另一个类型不适用的参数。

#### 3. 当前工程实际是怎么实现的

- 当前工程是单电机PMSM FOC。
- 未看到BLDC六步控制、ACIM控制内核和多电机控制内核切换实际接入。
- Hall相关头文件和备份文件存在，但当前Keil工程未编译Hall控制源文件。

#### 4. 哪种更好，为什么用现在这种

- 当前聚焦单电机PMSM FOC是更好的工程选择，符合当前硬件、RVDT反馈和MCSDK工程基础。
- 多内核平台化只有在产品确实要覆盖多种电机时才值得做。
- 当前不建议为了“大而全”引入BLDC/ACIM内核，否则会稀释PMSM主链路的验证资源。

## 五、高级性能与安全阶段

### 1. 温度线性降额

#### 1. 理论是怎么实现的

- 采集功率器件NTC、电机绕组PTC/PT100/PT1000或其他温度传感器。
- 温度低于第一阈值时允许满功率运行。
- 温度超过第一阈值后按线性或曲线方式降低Iq限幅、功率限幅或速度限幅。
- 温度超过第二阈值、传感器异常或温升斜率过快时进入故障停机。
- 降额状态应通过通信上报，并记录最高温度和触发原因。
- 线性降额可按双阈值实现：

$$I_{q\_limit} = I_{q\_rated}\left(1 - \frac{T_{meas}-T_{warn}}{T_{trip}-T_{warn}}\right)$$

- 温升斜率可在中频任务用离散差分计算：

$$\frac{dT}{dt} = \frac{T(k)-T(k-1)}{\Delta t}$$

#### 2. 理论需要注意哪些

- 过温保护只有停机阈值会造成产品体验差，也容易在边界工况频繁启停。
- 温度传感器安装位置和热滞后会导致芯片温度与测量温度不同步。
- 降额不能突变，否则等同于负载突卸或转矩突变。
- 温度开路/短路要单独诊断，不能误认为低温或高温。

#### 3. 当前工程实际是怎么实现的

- 母线电压和温度采样已接入MCSDK安全任务。
- 已看到过温保护路径。
- 未看到温度线性降额、温升斜率保护或双阈值降额策略。

#### 4. 哪种更好，为什么用现在这种

- 当前硬阈值过温保护简单可靠，适合早期保护功率器件。
- 产品级更好的方案是“线性降额 + 硬停机”，既保护硬件，也减少非必要停机。
- 建议保留现有硬停机作为最后保护，再在中频任务中增加Iq限幅降额层。
- 降额输出不要直接改PWM或电流采样链路，应进入统一限幅仲裁层，最终压缩速度环输出的`IqMax`或转矩模式目标上限。

### 2. 故障处理、BKIN与主动短路ASC

#### 1. 理论是怎么实现的

- 故障按严重程度分为可恢复告警、受控停机故障、立即关断故障和特殊安全动作故障。
- 普通过流、驱动故障应通过BKIN硬件快速封锁PWM。
- 过温、欠压、通信超时等可走软件状态机，按工况选择降额、STOP或FAULT。
- 高速永磁电机严重故障时，如果直接高阻关PWM会导致母线泵升，可执行主动短路ASC。
- ASC通常导通上三桥或下三桥之一，具体取决于功率级设计、驱动器能力和安全验证结果。
- 典型故障决策为：`严重故障 -> 判断速度是否高于基速/ASC阈值 -> 低速TURN_OFF_PWM -> 高速限时ASC -> 收敛到FAULT并等待人工复位`。

#### 2. 理论需要注意哪些

- ASC不是简单“任意三相短路”，必须验证电流峰值、器件SOA、驱动互锁和退出条件。
- 低速和高速故障策略可能不同，不能一个`TURN_OFF_PWM`覆盖全部。
- BKIN硬件封锁后软件状态机需要同步故障来源，否则恢复流程会混乱。
- ASC和母线过压保护、弱磁状态、机械速度必须联动。
- ASC动作必须通过示波器实测三相电流、母线电压和栅极波形，确认不会出现上下桥直通、驱动欠压或二次母线尖峰。

#### 3. 当前工程实际是怎么实现的

- MCSDK故障处理包含过温、过压、欠压、PWM/驱动故障、启动失败和FOC执行超时等路径。
- 当前BKIN引脚配置存在，但BreakState未启用。
- 当前过压策略主要是`TURN_OFF_PWM`。
- 未看到高速弱磁严重故障时强制导通上三桥或下三桥的ASC逻辑。

#### 4. 哪种更好，为什么用现在这种

- 对低速和普通故障，当前关PWM策略简单且安全，是合理默认行为。
- 对高速弱磁工况，更好的方案必须增加ASC，否则母线过压风险不可忽略。
- 当前先不用ASC可能是因为硬件SOA和动作时序尚未验证；但只要产品包含高速弱磁，ASC应进入高优先级重构项。

### 3. 预充电、主继电器与母线主动泄放

#### 1. 理论是怎么实现的

- 高压系统上电先闭合预充继电器，通过预充电阻给母线电容充电。
- MCU监测母线电压达到阈值后闭合主继电器，再断开预充继电器。
- 预充超时、电压不上升、继电器粘连、母线异常掉电都应触发故障。
- 停机后可通过泄放电阻/MOS或受控电机耗能策略降低母线残压。
- 预充流程建议按`INIT -> 闭合预充 -> Vbus达到目标比例 -> 闭合主继电器 -> 断开预充 -> READY`执行。若在超时窗口内Vbus不上升到目标比例，应判定预充失败。
- 主动泄放应在STOP或FAULT且转速安全后执行，目标是把母线电压降到维护安全阈值以下；泄放时间、泄放电阻温升和MOS SOA需要单独校核。

#### 2. 理论需要注意哪些

- 预充和MCSDK里的`CHARGE_BOOT_CAP`不是同一件事。前者是高压母线电容预充，后者多指栅极驱动自举电容充电阶段。
- 继电器控制必须考虑粘连检测和安全互锁。
- 主动泄放有热耗散风险，不能无限制导通。
- 高压残留需要满足安全法规和维修安全要求。

#### 3. 当前工程实际是怎么实现的

- 未看到高压预充继电器、主继电器、主动泄放MOS/电阻控制实现。
- 当前状态机中的`CHARGE_BOOT_CAP`是MCSDK自举电容充电状态，不等价于高压母线预充。

#### 4. 哪种更好，为什么用现在这种

- 如果当前硬件没有高压继电器和泄放电路，那么软件不实现是合理的。
- 如果产品面向高压母线或大电容系统，更好的方案必须增加预充和泄放控制，否则上电浪涌和残压安全不可控。
- 需要先确认硬件原理图，再决定软件是否补这一层。

### 4. 层次化状态机HSM

#### 1. 理论是怎么实现的

- 电机控制状态机通常包括INIT、CALIBRATION、READY、START、RUN、STOP、FAULT等主状态。
- 每个主状态可包含子状态，例如偏置校准、位置对齐、飞车检测、开环拉起、闭环切换。
- 故障态应具有最高优先级，能中断普通状态转移。
- 状态机要定义进入动作、退出动作、超时、失败回退和可接受命令。

#### 2. 理论需要注意哪些

- 状态机不能只靠散落的if判断，否则故障和模式切换很难维护。
- 校准、辨识、OTT、运行和参数保存之间必须互锁。
- 故障恢复不能自动重启到危险状态，应要求明确复位或满足恢复条件。
- 状态机状态应通过通信可读，便于上位机调试。

#### 3. 当前工程实际是怎么实现的

- MCSDK状态机已接入，包含`IDLE`、`OFFSET_CALIB`、`CHARGE_BOOT_CAP`、`ALIGNMENT`、`START`、`SWITCH_OVER`、`RUN`、`STOP`、`WAIT_STOP_MOTOR`、`FAULT_NOW`、`FAULT_OVER`等路径。
- SCC、OTT和通信寄存器已与部分状态和任务链路接入。
- RVDT故障、自动零位对齐、ASC、温度降额、预充泄放尚未形成完整统一HSM闭环。

#### 4. 哪种更好，为什么用现在这种

- 当前继续使用MCSDK状态机比另起炉灶更好，因为FOC、PWM、保护和接口都已经依赖它。
- 更好的做法是在MCSDK状态机外侧增加项目级HSM仲裁层，把RVDT、SCC、OTT、参数保存、ASC和应用模式纳入统一门禁。
- 不建议直接大改MCSDK核心状态机，优先通过封装和扩展降低重生成冲突。

## 六、系统支撑阶段

### 1. 通信、对象字典与上位机接口

#### 1. 理论是怎么实现的

- 调试阶段可使用UART、USB、ASPEP、MCP或自定义协议，实现高速变量读写和波形记录。
- 产品阶段若需要工业伺服兼容，通常接入CANopen、EtherCAT或其他现场总线。
- CANopen电机驱动应建立对象字典，映射控制字、状态字、目标速度、目标转矩、实际速度、故障码等对象。
- DS402状态机应与内部电机状态机做明确映射。
- 最小DS402对象映射可先规划如下，实际索引扩展时再和现有MCP寄存器对齐：

| Index | Name | 内部映射建议 |
| --- | --- | --- |
| `0x6040` | Controlword | 启停、故障复位、模式使能命令 |
| `0x6041` | Statusword | MCSDK状态机、故障、运行使能状态 |
| `0x6060` | Modes of operation | 速度、转矩、位置、回零模式 |
| `0x607A` | Target position | 应用层位置目标 |
| `0x60FF` | Target velocity | 速度环目标 |
| `0x6071` | Target torque | 转矩或Iq目标 |

#### 2. 理论需要注意哪些

- 调试协议和产品协议目标不同，不能把调试寄存器直接当量产对象字典。
- 运行态可写参数必须区分安全等级，保护阈值、硬件拓扑和对齐参数不应随意在线改。
- 通信解析不能阻塞FOC高频中断。
- CANopen/DS402引入后，状态机命令来源会变复杂，需要仲裁本地、上位机和总线命令。

#### 3. 当前工程实际是怎么实现的

- 当前UART + ASPEP + MCP通信已接入，支持Motor Pilot风格寄存器读写、同步/异步数据包和数据日志。
- 自定义寄存器覆盖RVDT、SCC/OTT、弱磁、多段PID、参数保存和性能统计。
- 上位机配置文件、QML和RegList JSON存在。
- CAN/CANFD/CANopen/DS402未接入，`HAL_FDCAN_MODULE_ENABLED`处于注释状态。

#### 4. 哪种更好，为什么用现在这种

- 对当前调试阶段，UART + ASPEP + MCP更好，因为它和MCSDK/Motor Pilot生态直接兼容，变量调试效率高。
- 对量产或工业现场，CANopen/DS402更好，因为主站兼容性、状态机规范和对象字典更标准。
- 现在使用ASPEP/MCP是合理的，但应提前规划对象字典映射，避免后续从调试寄存器迁移到DS402时大改结构。
- 推荐先做“对象字典设计文档”，再决定是否真正打开FDCAN和CANopen栈。这样不会影响当前Motor Pilot调试链路。

### 2. 强实时控制与低优先级任务隔离

#### 1. 理论是怎么实现的

- ADC注入转换完成或PWM更新中断触发FOC高频任务。
- 高频任务只做采样、坐标变换、PI、SVPWM、关键保护和必要的数据快照。
- 中频任务处理速度环、状态机、温度、母线、通信轻量处理和诊断。
- 低频或后台任务处理Flash保存、日志写入、参数校验、上位机大数据包解析等非强实时工作。
- 高低速任务之间通过双缓冲、原子变量或无锁队列交换数据。
- 推荐把任务边界写成工程规则：FOC ISR只允许固定时间算法和寄存器访问；中频任务允许状态机和滤波；后台任务才允许FlashDB事务、日志落盘和大包通信。

#### 2. 理论需要注意哪些

- Flash擦写、复杂字符串处理、阻塞通信、长循环滤波都不能进入FOC高频中断。
- 调试波形采样也要评估带宽，不能为了看数据破坏控制周期。
- 任何新增功能都要给出最坏执行时间预算。
- 中断优先级要明确，通信中断不能抢占FOC关键路径。
- 运行中在线参数修改建议先写RAM影子变量，停机或READY状态再集中落库。直接在RUN态触发Flash擦写是高风险设计。

#### 3. 当前工程实际是怎么实现的

- 当前FOC高频任务在PWM/ADC相关中断链路执行。
- MCSDK中频任务处理状态机、速度环、保护和通信。
- DAC调试输出、CPU性能测量和MCPA数据记录链路已接入高频任务。
- FlashDB/FAL已接入参数保存，但需要持续确认保存动作不进入强实时路径。

#### 4. 哪种更好，为什么用现在这种

- 当前MCSDK的高频/中频分层是正确架构，应继续沿用。
- 更好的工程化方向是为所有新增功能标注执行周期和任务归属，特别是RVDT读取、SCC、OTT、Flash保存和日志。
- 当前不引入复杂RTOS是合理的，因为裸机MCSDK调度已经满足单电机实时性；需要的是任务边界纪律，而不是更多线程。

### 3. 故障日志、黑匣子与参数追溯

#### 1. 理论是怎么实现的

- 普通故障日志记录故障码、子码、时间戳、状态机状态、控制模式和关键传感器值。
- 严重故障应记录故障前后一段高速数据切片，例如Id/Iq、Vd/Vq、速度、Vbus、温度、PWM占空比和状态机。
- 黑匣子数据先存RAM环形缓冲，故障后由后台任务写Flash。
- 日志应循环覆盖，并支持通过通信读取和清除。

#### 2. 理论需要注意哪些

- 故障中断里不能擦Flash。
- 日志写入不能破坏参数区。
- 黑匣子采样变量要少而关键，否则RAM和带宽很快耗尽。
- 日志需要绑定固件版本、参数版本和硬件版本，否则售后无法复盘。

#### 3. 当前工程实际是怎么实现的

- 当前FlashDB/FAL已接入参数保存。
- 未看到故障日志/TSDB接入，也未看到严重故障黑匣子写入FlashDB。
- MCSDK故障状态和通信寄存器可读取部分运行信息，但不是完整故障追溯系统。

#### 4. 哪种更好，为什么用现在这种

- 当前先做参数保存是合理的，因为控制参数比日志更早影响闭环可运行性。
- 产品级更好的方案必须补故障日志和黑匣子，尤其是弱磁、过压、RVDT故障和过流问题。
- 建议先实现低频故障摘要，再实现RAM高速环形缓冲，最后再做FlashDB持久化。

### 4. 系统验证与发布门禁

#### 1. 理论是怎么实现的

- 发布前应覆盖基础上电、PWM安全态、ADC采样、RVDT方向、偏置校准、相序对齐、空载闭环、负载阶跃、弱磁边界、故障注入和通信异常。
- 每项测试要记录固件版本、参数版本、硬件版本、电机型号、母线电压、散热条件和测试人员。
- 参数调整遵循单变量原则，所有曲线和参数快照可追溯。
- 保护功能必须有主动触发测试，而不是只看代码存在。
- 建议把台架门禁固化为五类测试：相序零位与空载梯度、负载突加突卸与PID整定、逆变效率Map、弱磁稳态边界、跨基速动态与急停过压拦截。

#### 2. 理论需要注意哪些

- 首次上电必须限流，且从低母线电压开始。
- 高速弱磁测试要有机械防护和母线能量处理能力。
- 负载突加突卸要从小阶跃开始，不能直接满载冲击。
- 测试不通过时要记录失败边界，而不是只记录成功点。
- 效率测试要区分直流输入功率、dq轴交流输出估算和测功机机械输出功率，避免变量缩放或符号约定导致效率结论失真。
- ASC和BKIN必须有单独的示波器证据，不能只依赖上位机低速刷新变量。

#### 3. 当前工程实际是怎么实现的

- 文档中已整理基础调试、空载运行、负载梯度、PID调试、效率测算、弱磁切入点、弱磁动态加减速和测试记录闭环等内容。
- 当前代码侧已具备上位机变量、MCP寄存器、DAC调试和性能统计支持，有利于测试记录。
- 但当前未看到完整自动化测试门禁、故障日志和发布包版本追溯闭环。

#### 4. 哪种更好，为什么用现在这种

- 当前先用人工台架测试加上位机记录是现实可行的，适合功能快速收敛。
- 更好的最终形态是形成固定测试矩阵、参数快照、波形模板和发布审查清单。
- 不建议等自动化完善后才测试；应边调试边沉淀测试项，逐步把高风险项固化为门禁。
- Release包建议至少包含固件镜像、参数黄金模板、寄存器/对象字典、五类台架测试报告、WCET时序记录、已知问题和回退方案。

## 七、应用层控制阶段

### 1. 位置闭环与S型轨迹规划

#### 1. 理论是怎么实现的

- 位置环位于速度环外侧，位置反馈来自RVDT、编码器或融合位置。
- 位置环通常输出目标速度，速度环再输出Iq参考。
- 轨迹规划器在位置指令前端生成平滑的位置、速度、加速度曲线。
- 高性能定位常用七段式S曲线，限制速度、加速度和Jerk。
- 位置控制应包含软限位、硬限位、跟随误差、回零和急停处理。

#### 2. 理论需要注意哪些

- 上位机或PLC下发的阶跃位置不能直接进入位置环。
- 位置环周期、速度环周期和反馈刷新周期要匹配。
- 轨迹中途改目标要保证输出连续，不能出现速度或加速度突跳。
- 位置误差过大时要按工艺选择降速、STOP或FAULT。

#### 3. 当前工程实际是怎么实现的

- 当前MCSDK中存在位置模式相关枚举和接口痕迹。
- 未看到实际位置环、S曲线轨迹规划、电子齿轮、电子凸轮等应用层实现。
- 当前工程主要是速度/转矩控制。

#### 4. 哪种更好，为什么用现在这种

- 当前先做速度/转矩控制是合理的，因为位置控制依赖底层电流、速度、RVDT和状态机稳定。
- 如果产品目标是伺服定位，更好的最终方案必须增加位置环和轨迹规划。
- 建议在速度闭环、弱磁和保护验证完成后，再把RVDT机械角度作为位置环反馈接入。

### 2. 工艺模式平滑切换

#### 1. 理论是怎么实现的

- 常见模式包括速度模式、转矩模式、位置模式、回零模式和工艺专用模式。
- 模式切换时应设置20ms到50ms过渡窗口，对旧模式输出和新模式目标做淡入淡出。
- 切入速度环时应预装载PI积分器，避免接管瞬间输出突变。
- 切换命令需要经过状态机、故障状态、速度范围、电流范围和参数权限检查。

#### 2. 理论需要注意哪些

- 不能简单断开上一级控制环，否则容易出现Iq尖峰和机械冲击。
- RUN态禁止修改硬件拓扑、保护阈值、相序零位等高风险参数。
- 切换失败必须保持原模式或进入受控STOP，不能进入半切换状态。
- 工艺模式需要记录命令来源和执行结果，方便追溯。

#### 3. 当前工程实际是怎么实现的

- 当前MCSDK支持速度模式和转矩模式基础接口。
- MCP寄存器可下发速度、转矩和部分控制命令。
- 未看到完整应用层模式平滑切换、输出交叉淡入淡出或PI积分器预装载机制。

#### 4. 哪种更好，为什么用现在这种

- 当前基础速度/转矩模式足够支撑底层调试，是合理起点。
- 对拧紧、压装、卷绕等工艺，更好的方案是增加模式切换过渡层，避免转矩冲击。
- 建议不要把工艺逻辑塞进FOC高频任务，而是在应用层生成平滑目标给MCSDK接口。

### 3. 电子齿轮与电子凸轮

#### 1. 理论是怎么实现的

- 电子齿轮通过主轴位置增量乘以齿轮比分子/分母，生成从轴目标位置。
- 电子凸轮通过主轴相位查表和插值，生成从轴目标位置或速度。
- 同步控制需要统一时间基准，可通过硬件捕获、SYNC、时间戳或高速总线实现。
- 齿轮比、凸轮表和同步状态应进入对象字典或参数系统。

#### 2. 理论需要注意哪些

- 主轴通信延迟、丢包和位置跳变会直接造成从轴冲击。
- 齿轮比和凸轮表必须有CRC、版本和边界模式。
- 同步误差超限时要定义降速、脱开同步、STOP或FAULT。
- 多轴同步对系统时间和通信确定性要求高，不适合随意叠加在未稳定的底层控制上。

#### 3. 当前工程实际是怎么实现的

- 当前未看到电子齿轮、电子凸轮、主轴捕获或多轴同步实现。
- 当前通信主线为UART ASPEP/MCP，不是实时多轴同步总线。

#### 4. 哪种更好，为什么用现在这种

- 当前不实现电子齿轮/凸轮是合理的，因为工程目标仍集中在单轴电机驱动核心能力。
- 如果后续产品需要多轴同步，更好的方案是先确定现场总线和时间同步机制，再设计应用层。
- 不建议在当前UART调试协议上强行实现高精度同步控制。

## 八、建议重构优先级

### 1. 第一优先级：安全闭环

- 启用并验证BKIN硬件刹车。
- RVDT DOS/LOT故障进入MCSDK故障状态机。
- 增加电流采样残差诊断和ADC饱和诊断。
- 明确高速弱磁故障策略，评估并实现ASC。

### 2. 第二优先级：参数闭环

- 补齐FlashDB中弱磁、前馈、RVDT零位、温度降额和保护阈值保存。
- 增加参数版本、CRC、默认值、范围检查和恢复机制。
- 增加SCC/OTT整定结果确认、保存和回退。

### 3. 第三优先级：性能闭环

- 完成弱磁稳态边界和动态跨基速测试。
- 在参数可信后分阶段启用前馈解耦。
- 评估死区补偿和逆变器非线性标定。
- 根据电机类型决定是否引入MTPA/MTPV。

### 4. 第四优先级：产品化闭环

- 增加故障日志和黑匣子。
- 规划CANopen/DS402或目标产品通信协议。
- 增加位置环、S曲线和工艺模式平滑切换。
- 建立固定测试矩阵和发布门禁。
